---
title: "Refresh a Nova Guideline Report"
sidebarTitle: "Refresh Report"
description: "Re-run a guideline report for a Nova release, optionally overriding scope or forcing fresh evidence collection. Returns a new report ID immediately."
---

Use this endpoint to regenerate a guideline report — for example, after resolving outstanding tickets, merging a required MR, or correcting the component scope. Refreshing queues a new background job and returns a new report ID immediately. You can override any of the original request parameters; fields you omit are inherited from the original report. The new job progresses through the standard `queued` → `running` → `done` lifecycle, which you can track using [Get Report](/api/get-report).

**Endpoint**

```
POST https://redp.app/api/v1/guideline-reports/:id/refresh
```

## Path Parameters

<ParamField path="id" type="string" required>
  The ID of the report you want to refresh. This can be any existing report — `done`, `failed`, or even `cancelled`.
</ParamField>

## Request Body

<ParamField body="projectId" type="string">
  Override the project ID for the new report. If omitted, inherited from the original report.
</ParamField>

<ParamField body="releaseId" type="string">
  Override the release ID for the new report. Required if `version` is not provided and you want to change the target release. If omitted, inherited from the original report.
</ParamField>

<ParamField body="version" type="string">
  Override the release version string for the new report. Required if `releaseId` is not provided and you want to change the target release. If omitted, inherited from the original report.
</ParamField>

<ParamField body="scope" type="string">
  Override the component scope. Accepted values: `all` or `custom`. If omitted, inherited from the original report.
</ParamField>

<ParamField body="componentIds" type="array">
  An array of component ID strings. Required when `scope` is `custom`; ignored when `scope` is `all`.
</ParamField>

<ParamField body="forceRefresh" type="boolean">
  When `true`, redp discards all cached evidence and re-collects data fresh from source systems. Defaults to `false`, which reuses any evidence that was successfully cached during the original run.
</ParamField>

<ParamField body="generatedBy" type="string">
  A user identifier recorded in the new report for audit logging purposes.
</ParamField>

## Example Request

```bash
curl -X POST https://redp.app/api/v1/guideline-reports/rpt_01hx4k9m2n3p5q7r/refresh \
  -H "Authorization: Bearer <your-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "forceRefresh": true,
    "generatedBy": "jane.doe@example.com"
  }'
```

## Response — 202 Accepted

A new report job has been created. Use the returned `id` to poll for the result.

```json
{
  "id": "rpt_01hy5l0n3o4q6r8s",
  "status": "queued",
  "projectId": "nova-platform",
  "releaseId": "rel-2024-11-0",
  "version": "2024.11.0",
  "scope": "all",
  "createdAt": "2024-11-14T11:05:00Z"
}
```

<ResponseField name="id" type="string">
  The ID of the newly created report job. This is different from the original report ID.
</ResponseField>

<ResponseField name="status" type="string">
  Initial job state. Always `queued` on creation.
</ResponseField>

<ResponseField name="projectId" type="string">
  The project ID used for this report run.
</ResponseField>

<ResponseField name="releaseId" type="string">
  The release ID used for this report run.
</ResponseField>

<ResponseField name="version" type="string">
  The release version string used for this report run.
</ResponseField>

<ResponseField name="scope" type="string">
  The component scope used for this report run: `all` or `custom`.
</ResponseField>

<ResponseField name="createdAt" type="string">
  ISO 8601 timestamp recording when the new report job was created.
</ResponseField>

<Note>
  Refreshing a report creates a new, independent report record. The original report is retained in full and remains accessible by its ID, so you can compare the two runs side by side.
</Note>

<Tip>
  Use `forceRefresh: true` after rotating an access token or fixing an integration permission issue to ensure previously failed evidence fetches are retried rather than skipped.
</Tip>

## Error Responses

| Status | Meaning |
|--------|---------|
| `400 Bad Request` | The request body contains invalid values. |
| `401 Unauthorized` | The `Authorization` header is missing or the token is invalid. |
| `404 Not Found` | No report exists with the given ID. |
